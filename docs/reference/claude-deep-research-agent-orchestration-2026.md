# Multi-agent AI orchestration for autonomous software delivery

**Your architecture is sound in its broad strokes but harbors three design assumptions that production evidence contradicts.** The artifact-based Git handoff, DAG-based workflow, and discrete session-per-stage choices all align with emerging best practices. But the flat topology of 30+ agent roles, the absence of a durable execution engine beneath Claude Code SDK, and the treatment of AI validation gates as deterministic CI steps will each cause operational failures. This report presents evidence from 150+ production systems, academic papers, and documented implementations from 2024–2026 across all eight requested topics, culminating in six actionable decisions.

The field moved fast between 2024 and 2026. Google DeepMind published the first rigorous scaling science for multi-agent systems in December 2025. Anthropic renamed Claude Code SDK to "Claude Agent SDK" in September 2025, validating the general-purpose substrate bet. The MAST taxonomy gave us the first empirical failure classification for multi-agent systems, finding **41–86.7% failure rates** across production frameworks. And Gartner predicted that over **40% of agentic AI projects will be canceled by 2027** due to escalating costs and inadequate controls. The evidence is now sufficient to make firm architectural recommendations.

## 1. What actually works in multi-agent orchestration

The orchestration landscape has consolidated around three tiers. **LangGraph** leads in adoption at 34.5 million monthly downloads and production deployments at AWS and Ally Tech Labs. **Temporal** is the definitive durable execution engine, used by Snap for millions of monthly workflows and by Retool as foundational infrastructure. **CrewAI** powers PwC's code generation pipeline, which achieved a **7x accuracy improvement** (10% to 70%) through structured multi-agent architectures. AutoGen has been merged into Microsoft's Agent Framework (public preview October 2025) and is now in maintenance mode. OpenAI's Agents SDK, released March 2025, supports handoffs and tool calling but lacks native fan-out—the community has requested this capability without resolution.

The most important finding for your architecture: **Temporal should be your orchestration backbone, not a custom DAG engine.** Temporal separates deterministic workflow orchestration from non-deterministic activity execution (LLM calls, tool use). On crash recovery, it replays event history so an agent resumes exactly where it failed—saving both tokens and money versus re-running from scratch. This directly solves your mid-stage resume requirement. Retool's team stated explicitly: "It just wouldn't have been possible without Temporal." Temporal also launched a public-preview integration with OpenAI's Agents SDK in 2025, developed in collaboration with OpenAI, demonstrating that the pattern of wrapping LLM agent calls as Activities within durable Workflows is becoming standard.

Your file-based Git handoff has strong precedent in the **blackboard architecture** pattern. Fast.io's 2025 documentation reports systems using shared blackboards show **35% processing time reduction** and fewer hallucination loops compared to streaming through API layers. The Lore protocol (arXiv, March 2026) goes further, repurposing git commit trailers as structured knowledge channels between agents—the first formal proposal to use git's native metadata as an inter-agent communication protocol. Anthropic's own long-running agent harness uses git commits plus a progress file as the canonical state mechanism across sessions. The evidence supports your choice, with one caveat: git-based handoff introduces polling latency, so time-critical coordination between parallel agents should use in-memory signals (Temporal Signals) while preserving artifacts in git for the audit trail.

Fan-out/fan-in with contradictory outputs is the hardest unsolved problem in your pipeline. Skywork.ai's production recommendation is explicit: **cap parallel agents at 3–5, because beyond that, merge complexity eats the gains.** The most effective merge pattern combines structure-first normalization (deduplicate via similarity checks) with a reviewer agent that resolves conflicts and triggers targeted re-retrievals. Google's ADK provides a native `ParallelAgent` plus `MergerAgent` pattern. Research from EMNLP 2025 found that implicit consensus—where agents observe others' outputs without formal voting—consistently outperformed explicit consensus mechanisms. The dangerous finding: multi-agent debate research (AgentAuditor, February 2026) demonstrated that majority voting can produce **"confabulation consensus"** where multiple agents repeat the same incorrect intermediate claims, requiring auditing at divergence points rather than frequency-based selection.

### Silent killers nobody talks about

The most insidious production failure is **context pollution at scale.** A first-hand production report from Bijit Ghosh (February 2026) documents the inflection point: "Problems started when we crossed into double digits of agents. Context stopped being an asset and started behaving like technical debt." The system only stabilized when agents were made stateless and short-lived, with all global reasoning moved into the orchestration layer. The cost profile is O(n²)—a few hundred KB per agent silently multiplies into MB-scale payloads per request.

**Cascading hallucination** is the second killer. OWASP ASI08 (2025–2026) documents it formally: "Errors persist in agentic AI memory, context windows, and shared knowledge bases. Unlike stateless requests, agentic AI errors can contaminate future reasoning cycles, turning transient errors into permanent behavioral drift." Galileo.ai documented a production incident where an inventory agent invented a nonexistent SKU, then called four downstream APIs to price, stock, and ship the phantom item. By the time monitoring caught it, four systems were poisoned.

**Agent laziness** emerges predictably past 5+ agents. Ghosh's report: "Past 5+ agents, ownership diffuses. Agents optimize for safe micro-changes, abandon hard cases, and recreate human meeting dynamics in silicon." EMNLP 2025 research confirmed that stubborn agents create leader-follower dynamics where the entire system converges on the most assertive agent's output rather than the best one.

## 2. Claude Agent SDK as a universal substrate is validated but fragile

Anthropic renamed Claude Code SDK to **Claude Agent SDK** in September 2025, explicitly positioning it as a general-purpose agent runtime. Their engineering blog stated: "Claude Code has become far more than a coding tool. We've been using it for deep research, video creation, and note-taking. It has begun to power almost all of our major agent loops." This validates your architectural bet. No other tool in the ecosystem—not Cursor, Devin, OpenHands, SWE-Agent, or Aider—has made this pivot. Builder.io's 2026 comparison confirmed: "Claude Code is agent-first. You describe what you want, the AI drives, and you review. Cursor is IDE-first."

The "one substrate, many roles" pattern has theoretical support from Andrew Ng, who explains that "an agentic workflow in which the LLM is prompted to focus on one thing at a time can give better performance" even when calling the same model. The key mechanism is **attentional focus**—a role-specific system prompt narrows the solution space and improves output quality. However, Yoav Goldberg cautions that the distinction between "multi-agent" and "single-agent multi-role" is often arbitrary, and LLMs still struggle with "very long, multi-objective prompts" and "a very large set of tools."

Your 43 cognitive skills have direct academic support. The **Cognitive Tools** paper (arXiv, June 2025) found that modular prompt compositions consistently outperformed monolithic prompting, with improvements ranging from **+4.2% to +27.2%** across models. A context engineering survey documented that GPT-4.1 performance on AIME2024 increased from 26.7% to 43.3% through structured cognitive operation sequences. The mechanism is reduced interference—each cognitive operation runs in isolation, reducing context noise. Critically, **progressive disclosure is mandatory**: Anthropic's own Skills feature loads specialized prompts on-demand rather than all at once. PromptForward confirmed that skill definitions consume tokens on every turn if pre-loaded, while on-demand loading fits comfortably under a few hundred tokens and is only read when invoked. Without progressive disclosure, 43 skills at ~300–500 tokens each means **13K–22K tokens consumed on every turn**, directly reducing reasoning capacity.

### The rate limit trap

The SDK's rate limiting is the most dangerous operational risk for your substrate choice. A single "edit this file" command can consume **50K–150K tokens**. Fifteen iterative commands can push final request context to 200K+ input tokens. Hidden background operations—conversation indexing, context management, tool-use overhead—consume tokens invisibly. February 2026 Hacker News threads documented rate limits triggering without corresponding visible usage. For a 7-stage pipeline where each stage might make 8–12 API calls, you could exhaust rate limits partway through a single experiment. The mitigation: route through **AWS Bedrock** (which you're already planning) rather than the direct Anthropic API, using Application Inference Profiles with per-experiment rate limit allocation. Bedrock's usage-based pricing avoids the per-seat rolling window limits of the direct API.

Context rot is empirically confirmed by every study. **Chroma Research** (2025) tested 18 frontier models and found that every single model gets worse as input length increases, at every increment tested. Cognition (Devin) measured that by 35 minutes of agent operation, every agent's success rate drops, and **doubling task duration quadruples the failure rate.** Your choice of separate sessions per stage with artifact handoff is strongly validated. Anthropic's own recommendation is the two-part pattern: initializer agent for first session, coding agent for each subsequent session, with structured handoff via progress files plus git history.

## 3. Thirty-plus roles is too many without hierarchical decomposition

Google DeepMind's landmark study (December 2025) across 180 agent configurations provides the definitive evidence. On parallelizable tasks, centralized coordination improved performance by **+80.9%** over single agents. On sequential tasks, every multi-agent variant **degraded performance by 39–70%.** The critical finding is the **"45% threshold rule"**: multi-agent coordination yields returns only when single-agent baseline accuracy is below 45%. If the base model already hits 80% on a task, adding agents introduces more noise than value.

The **"17x error trap"** analysis (January 2026) of DeepMind's data found that unstructured multi-agent networks amplify errors up to **17.2x** compared to single-agent baselines. This is not catastrophic failure—it is "quiet compounding of small errors that produces confident nonsense." The analysis proposes that most complex multi-agent systems can be decomposed into **10 fundamental archetypes**: Orchestrator, Planner, Executor, Evaluator, Synthesizer, Critic, Retriever, Memory Keeper, Mediator, Monitor. These should be organized into functional control planes (Plan-Do-Verify) rather than as a flat collection.

**Practitioner consensus from DEV Community (2026) is unambiguous: "3–7 agents work best for most workflows. Below three, you're probably fine with a single agent. Above seven, coordination complexity outweighs benefits unless you use hierarchical structures with team leaders managing subgroups."** MetaGPT validated specialization with 85.9% Pass@1 using 5 roles, and ChatDev used 7 roles effectively—but ChatDev's communication costs exceeded $10 per HumanEval task. Your 30+ flat roles will fall into the "bag of agents" anti-pattern.

The restructuring recommendation supported by evidence:

- **Generation agents** (discipline-based): Frontend Engineer, Backend Engineer, DBA—these produce code and should be organized as 2–3 team hierarchies of 3–5 agents, each under a team lead agent
- **Review agents** (concern-based): Security Auditor, Accessibility Auditor, Performance Auditor—these evaluate outputs and should operate as independent validators in a Verify plane
- **The mediator works but is a bottleneck**: Sarkar et al. (May 2025) documented mediators achieving sub-100ms turnaround with 15% fewer false positives, but they introduce single points of failure. Your Design Mediator should have a fallback—if mediation fails, escalate to human review rather than blocking

The authority parameter concept has formal support. Feng, McDonald & Zhang (July 2025) published a five-level autonomy framework ranging from L0 (human does everything) to L4 (fully autonomous with monitoring). The critical insight: **"Autonomy is a design decision that does not need to be tightly coupled with agent capability."** Production implementations use confidence-based escalation: financial services at 90–95% thresholds, routine tasks at 80–85%, high-stakes at 95%+. Illumination Works (December 2025) describes the pattern where "the AI decides when to involve the human" by reasoning: "I am 95% confident this is safe, but the stakes are high, so I'll escalate."

## 4. AI validation gates need the "triple-run" pattern to survive non-determinism

The non-determinism problem is worse than you think. **Even temperature=0 does not guarantee deterministic output.** Mira Murati's Thinking Machines Lab identified **batch size variability**—not floating-point arithmetic—as the primary cause. They demonstrated perfect determinism using batch-invariant kernels, but this is not widely deployed. FlowHunt (2026) confirmed: "Run the same prompt ten times with temperature=0, and you get different results." For CI gates, this means a change that passes all 8 gates on Monday might fail Gate 5 on Tuesday with the same code.

First-generation AI code review tools had a **48% false positive rate** (cubic.dev, 2025 benchmarks). Current tools claim 11% but this is self-reported. CodeRabbit's December 2025 analysis of 470 PRs found AI-generated code produces **1.7x more issues** than human code, with 1.4x more critical and 1.7x more major issues. The trust threshold is documented: a tool catching 60% of bugs with 40% false positives is worse than one catching 45% with 10% false positives, because developers ignore the noisy tool entirely, making its effective catch rate zero.

The production-viable pattern for AI gates combines three mechanisms:

- **Self-consistency sampling**: Run each gate 3 times with semantic prompt variations, take consensus. Research shows majority voting reduces maximum error from 17.6% to **4.8% after data repair** (arXiv, 2025). But AgentAuditor warns that simple majority voting can produce "confabulation consensus"—audit divergence points, not frequencies
- **Structured output schemas**: Force each gate to produce typed JSON with confidence scores, evidence citations, and severity classifications. FINOS AI Governance Framework: "Determinism is something you impose with contracts around it"
- **Layered evaluation**: Deterministic checks (linting, type checking, WCAG rule-based scanning) run first on every commit. LLM evaluation runs second on push to main. Human judgment is the final layer. A SEC filing analysis team found this layered approach took reliability from ~60% to **98%+**, though cost per analysis went up 4x

**Semantic accessibility auditing is research-only.** AccessGuru (ACM SIGACCESS 2025) achieved **84% violation score decrease** combining traditional tools with LLMs, and López & Pereira (2024) showed LLMs outperform traditional tools on manual-check criteria like non-text content and link purpose. But W4A 2025 found LLMs produce "technically correct but practically inadequate solutions" for complex ARIA implementation. No production CI deployment of LLM-based semantic accessibility auditing was found. Plan for a hybrid: axe-core for deterministic rule checks, LLM auditor for semantic review, human spot-check for ARIA and screen reader testing.

Adversarial AI testing in CI has real tooling. **Garak** (open-source LLM vulnerability scanner) integrates via GitHub Actions. **DeepTeam** implements 40+ vulnerability classes aligned to OWASP LLM Top 10. **SuperClaw** (late 2025) outputs SARIF format for GitHub Code Scanning integration. However, these tools test AI model outputs (chatbots, RAG responses), not AI-generated code. The pattern of "agents that attack their own code output" remains largely conceptual. The closest production system is the DEV Community's documented 8-gate validation system, where each flagged issue goes through a separate validation agent that confirms the issue actually exists—filtering false positives before reporting.

## 5. AI-generated code is faster to produce and measurably worse

The evidence is now overwhelming and quantified. **Veracode's 2025 report found 45% of AI-generated code contained OWASP Top 10 vulnerabilities.** Java had the worst rate at 72% failure. XSS defense failed 86% of the time. Log injection defense failed 88%. Larger models performed no better—the problem is systemic. DryRun Security (March 2026) tested Claude Code, OpenAI Codex, and Gemini building applications from scratch: **87% of the 30 PRs contained at least one vulnerability**, producing 143 total security issues including insecure JWT defaults, missing brute-force protection, and CSRF vulnerabilities.

Apiiro's Fortune 50 enterprise research (September 2025) found AI-generated code introduced **10,000+ new security findings per month** by June 2025—a 10x spike from December 2024. The curve is not flattening. The "Speed at the Cost of Quality" Cursor study (arXiv, November 2025) used a difference-in-differences design across 807 repos and found static analysis warnings increased **30%** and code complexity increased **41%** post-adoption. Initial 3–5x velocity gains **dissipated after two months**, with technical debt then reducing future velocity in a self-reinforcing degradation cycle.

For your architecture, this means **automated security scanning on every agent commit is non-negotiable**, not just on the final output. Design tokens as constraints work—Romina Kavcic documented that "when Claude generates code, it cannot skip tokens; every visual property resolves to a value, making tokens your guaranteed point of influence." But attention dilution causes drift: design token rules stated early in a session lose influence as context grows. The mitigation is re-injecting token references at the start of each stage and enforcing them via linting.

The revision loop has documented failure modes. The **"fix-fix spiral"** is the most dangerous: Claude introduces an error, tries to fix it, introduces a new error, and the code gets progressively worse with each iteration. The "loop" variant: same fix, same error, slightly different approach for three iterations. The convergence guidance: **TDD as convergence mechanism** (tell the agent to iterate until tests pass, with a hard cap of 5 iterations) converges faster than human-in-loop correction. A 50-iteration autonomous loop on a large codebase can cost **$50–100+ in API credits**. The METR study (July 2025) found that experienced developers actually took **19% longer** using AI tools, despite perceiving 20% faster productivity.

The AGENTS.md convention, adopted by over **60,000 open-source projects**, is the most important missing piece in your architecture. It encodes coding standards, file structure, naming conventions, and architecture boundaries in a machine-readable format that agents load at session start. OpenAI, Sourcegraph, and Google announced collaboration to formalize the standard in July 2025. Factory.ai's approach is the gold standard: "Agents write the code; linters write the law." Every observed style violation becomes an executable constraint.

## 6. Cost attribution requires a gateway layer you haven't designed

Enterprise AI agent deployments average **340% cost overruns** above initial estimates, and 73% of development teams lack real-time cost tracking. Mavvrik's survey of 372 enterprises found **80% miss AI infrastructure forecasts by more than 25%.** Pre-execution cost prediction for multi-agent workflows remains an unsolved production problem—the best approach is historical profiling combined with hard budget caps.

The hidden cost multipliers will surprise you. Complex agents with tool-calling consume **5–20x more tokens** than simple chains. A recursive agent loop documented in production made 47,000 API calls in 6 hours at $0.03/call, burning **$1,410 on a single stuck process**. Context window padding waste is endemic: one customer service agent maintaining 20,000 tokens of history per interaction reduced to 3,500 tokens (82% reduction) through smart summarization. Reasoning tokens in Claude's extended thinking can multiply output costs by **10–30x** for complex tasks, and these costs are invisible in responses without instrumentation.

**AWS Bedrock's native token tracking is inadequate for your needs.** Standard billing shows costs aggregated by model and region only—no attribution to users, experiments, stages, or agents. Application Inference Profiles provide cost allocation tags that appear in Cost Explorer, but per-request granularity requires response header parsing (`x-amzn-bedrock-input-token-count`). AWS provides no native IAM policies to limit Bedrock API calls or token consumption.

The solution is a **gateway layer**. LiteLLM provides a 5-level hierarchical budget system (Organization → Team → User → Key → End User) with hard budgets that reject requests and soft budgets that alert only. Portkey routes 400B+ tokens daily with circuit breaker patterns. RouteLLM (UC Berkeley/LMSYS) achieved **85% cost reduction** on MT-Bench while maintaining 95% quality by routing between expensive and cheap models. BudgetMLAgent demonstrated **96.4% cost reduction** using Gemini-Pro as the first cascade level before escalating to GPT-4. For philanthropic money, model routing is not optional—it is the single highest-leverage cost optimization available.

The recommended budget escalation pattern from production deployments: 60% threshold triggers owner alerts, 85% triggers escalation with grace window, **100% is a hard cap** that disables non-critical paths and notifies finance.

## 7. Git as institutional memory works but needs provenance tooling

The **git-ai** tool (December 2025) is the most directly relevant production system. It fingerprints AI-generated code at creation time, linking every line to the agent, model, and session that generated it via Git Notes. A Fortune 500 fintech discovered 34% of their codebase was AI-generated during a pre-IPO audit using this tool. The **Lore protocol** (arXiv, March 2026) formalizes git trailers as structured knowledge channels, with vocabulary including `Lore-intent:` (why the change was made), `Lore-rejected:` (alternatives considered), and `Lore-constraint:` (constraints that shaped the decision). These are queryable via standard `git log --trailer=` commands.

Anthropic's own internal pattern validates your approach: agents commit progress to git with descriptive messages, use `claude-progress.txt` plus git history as institutional memory, and run `git log --oneline -20` to orient at session start. GitHub's hard limit of **100,000 repositories per organization** (enforced April 2025) sets the ceiling. At one repo per experiment, this is comfortable for years of operation—but individual repos should stay under 1 GB, with Git LFS for binary artifacts.

The institutional learning opportunity is quantified. QMatSuite (arXiv, March 2026) demonstrated that accumulated knowledge from past experiments reduced reasoning overhead by **67%** and improved accuracy from 47% to 3% deviation. The critical insight: knowledge systems should encourage **principle-level entries over numerical recipes**—agents that simply copy parameters from past successes without understanding applicability fall into an anti-pattern. All 41 knowledge patterns in QMatSuite were produced exclusively in dedicated review sessions, not during execution, suggesting you need a separate "learning extraction" phase after experiment completion.

The operational failures to anticipate are documented from a study of **33,596 agent-authored PRs**: 38% suffered reviewer abandonment (no meaningful human review), **23% were duplicates** (agent unaware of existing work), and 17% broke CI. The duplicate rate is alarming for your architecture—if multiple experiments run similar prompts, agents will produce overlapping work without coordination. Additionally, **77.5% of merged agentic PRs are self-merged**, raising governance concerns for a philanthropic platform that requires accountability.

## 8. Observability is the weakest link in the entire ecosystem

The Cleanlab survey of 1,837 engineering leaders (2025) found that **fewer than 1 in 3 teams are satisfied with agent observability solutions**, making it the weakest link in production agent systems. 63% plan to improve observability as their #1 investment priority. The field is so immature that **70% of regulated enterprises rebuild their AI agent stack every 3 months or faster.**

Automated failure attribution—determining which agent caused a pipeline failure—is an **unsolved research problem**. The Who&When dataset (ICML 2025 Spotlight) achieved only **53.5% accuracy** identifying the failure-responsible agent, and a mere **14.2% accuracy** pinpointing the specific failure step. Even SOTA reasoning models (o1, DeepSeek R1) failed to achieve practical usability. For a 7-stage pipeline with 30+ agents, this means debugging will be predominantly manual.

The cascading failure problem is formally characterized. AgentAsk (arXiv, 2025) documents **"chainwise amplification"** where small inconsistencies accumulate into system-level failures. The Gradient Institute's Australian government report illustrates: a demand forecasting agent makes a small error, which the procurement agent builds on, which production planning amplifies, which logistics compounds—"the final output is nonsensical or dangerously incorrect." Independent judge agents with isolated contexts at each handoff reduce cascading failures by 40%+ (Augment Code, PwC evidence).

The critical observability gap nobody mentions: **agents don't throw 500s when they fail.** OneUptime (March 2026) documented a production incident where an AI agent autonomously scaled down a critical database cluster based on traffic patterns, killing production for 11 hours—"nobody was watching." Traditional alerting assumes errors manifest as exceptions. Agent failures manifest as **confident wrong answers with 200 OK status codes.** Your observability must evaluate output quality, not just execution success.

The recommended tool stack based on evidence: **LangSmith** for development-time tracing (near-zero performance overhead), **Langfuse** or **Helicone** for production cost and quality monitoring (open-source, self-hostable), and **OpenTelemetry GenAI semantic conventions** as the underlying standard to prevent vendor lock-in. Separate platform metrics (agent latency, token costs, retry rates) from application metrics (output quality, task completion, user satisfaction)—the AgentSight paper (August 2025) formalizes this as the "semantic gap" that existing tools fail to bridge.

## Synthesis: six decisions, not summaries

### Three architectural decisions this research would change

**1. Add Temporal as the durable execution layer beneath Claude Agent SDK.** Your current design has no mechanism to resume a failed stage without re-running completed stages. Temporal provides exactly this: deterministic workflow orchestration wrapping non-deterministic LLM activities, with automatic state persistence and mid-stage resume. Each of your 7 stages becomes a Temporal Activity; the DAG becomes a Temporal Workflow. Claude Agent SDK sessions run inside Activities. This also gives you built-in rate limiting, retry with backoff, circuit breakers, and time-travel debugging—all of which you would otherwise build from scratch. Retool and Snap prove this works at scale.

**2. Restructure 30+ flat roles into 3 hierarchical teams of 3–5 agents each, plus an independent validation plane.** Evidence from DeepMind, the 17x error trap analysis, and practitioner consensus converges: above 7 agents in a flat topology, coordination complexity destroys value. Reorganize into: a Design team (Product Designer + Visual Designer + Architect under a Design Lead), a Build team (Frontend + Backend + DBA under a Build Lead), and a Validation plane (Security + Accessibility + Architecture reviewers operating independently). The Design Lead replaces your Mediation agent as the merge point. The validation plane runs concern-based reviewers in parallel with an independent meta-validator that confirms flagged issues actually exist before blocking.

**3. Deploy a gateway layer (LiteLLM or Portkey) between your orchestrator and all LLM APIs.** AWS Bedrock's native cost tracking is aggregated by model and region, providing zero per-experiment, per-stage, or per-agent attribution. A gateway gives you hierarchical budgets, model routing for cost optimization (expected 40–85% savings), circuit breakers for runaway loops, and the token-level observability that philanthropic accountability requires. This is not optional infrastructure—it is the difference between controllable and uncontrollable costs.

### Two biggest unsolved problems

**1. Pre-execution cost prediction for multi-agent workflows.** 80% of enterprises miss AI forecasts by more than 25%. No production system can reliably predict what a 7-stage, 30-agent experiment will cost before running it. The best available approach is historical profiling by prompt complexity class combined with hard budget caps—but this means you cannot tell a center leader "this experiment will cost approximately $X" before they submit it. The DAAO framework (2025) proposes a VAE-based difficulty estimator, but it has not been deployed in production.

**2. Automated failure attribution across multi-agent pipelines.** When an experiment produces a broken application, identifying which agent in which stage caused the failure achieves only 53.5% accuracy with current methods. Pinpointing the specific failure step drops to 14.2%. For non-technical staff who submitted a prompt and received a broken result, the system cannot reliably explain what went wrong. This means every failure requires manual investigation by an engineer who can read agent transcripts across multiple stages.

### The most dangerous assumption

**That AI validation gates will behave deterministically enough to serve as CI promotion criteria.** Your architecture treats Accessibility Auditor, Security Auditor, and Architecture Reviewer as pipeline gates that block or pass promotion. But the same audit run twice will produce different results. Temperature=0 does not solve this—batch size variability is the root cause, and the fix (batch-invariant kernels) is not widely deployed. A triple-run consensus pattern reduces variance significantly but triples cost and latency per gate. Worse, cascading non-determinism across 8 gates means a change that passes all gates Monday may fail Tuesday. The mitigation is to layer deterministic checks (linting, type checking, axe-core) as hard gates, and treat LLM-based gates as advisory with confidence scores—blocking only on high-severity, high-confidence findings from consensus runs.

### The failure mode nobody is talking about

**Experiment cross-contamination through shared infrastructure.** Your architecture serves two organizations sharing Neon PostgreSQL, Vercel, AWS Bedrock, and a design token library. When multiple experiments run concurrently, they share rate limits, model API quotas, database connection pools, and Vercel deployment slots. At 10 experiments/month this is invisible. At 100/month, Experiment A's token-heavy Research stage will throttle Experiment B's Build stage mid-deployment. Shared Neon connection pools will hit limits. Vercel's concurrent build queue will serialize experiments during peak hours. The India-based organization's morning work hours overlap with US-based overnight batch processing. You need per-experiment resource isolation: dedicated Bedrock usage quotas via Application Inference Profiles, per-experiment Neon database branches (which Neon natively supports), and deployment queue management with priority tiers. Without this, your first scaling incident will be one organization's experiment degrading the other's—politically catastrophic for a shared philanthropic platform.

### The single most important production system to study

**Anthropic's own multi-agent research system**, documented in their engineering blog. It uses a lead agent that analyzes queries, develops strategy, and spawns subagents—each acting as an "intelligent filter" with isolated context. It faces identical challenges to your architecture: context management across agents, merge of parallel outputs, quality degradation over long sessions, and the need for outcome-based evaluation rather than process-based. Their key insight maps directly to your design: "Even with identical starting points, agents might take completely different valid paths to reach their goal." Their published solutions—the initializer/coding agent pattern, progress files, git-based state, and compaction management—are the closest validated production analog to what you are building. Pair this with study of **Retool's Temporal-based agent architecture** for the durable execution patterns you need.

### The question this prompt didn't ask

**How will you prevent the system from training non-technical staff to accept mediocre output?** Every production study shows AI-generated code is measurably worse than human-written code: 1.7x more issues, 45% containing OWASP vulnerabilities, 30% more static analysis warnings, complexity increasing 41%. The METR study found developers perceived 20% faster productivity while actually being 19% slower. Your revision loop—staff reviews, leaves comments, agents revise—assumes staff can recognize quality problems in a live web application. But non-technical monastics and administrators lack the mental model to identify subtle accessibility failures, security vulnerabilities, or architectural debt. They will approve applications that look correct but are structurally unsound. Over time, the platform will produce a portfolio of applications with accumulating technical debt, security vulnerabilities, and accessibility gaps that nobody notices until an audit or incident. **You need an independent quality review by a technical human—not for every experiment, but as a sampled audit layer—before applications enter long-term production use.** The agents are the builders; the gates are the inspectors; but you need a building code inspector who answers to neither.
